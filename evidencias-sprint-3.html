1. Objetivos del Sprint 3

Durante este sprint el equipo avanzo sobre la parte funcional CORE de SalesGuard para pasar de una base de autenticacion a un flujo real de carga, almacenamiento y analisis de ventas.

  • Registrar datasets de ventas asociados a un usuario.
  • Guardar ventas diarias por fecha dentro de cada dataset.
  • Ejecutar un analisis estadistico simple con z-score.
  • Persistir anomalias detectadas dentro de la base de datos.
  • Enviar una notificacion por email o mostrar un modo demo claro.
  • Actualizar Swagger, README y artefactos de presentacion.
El sprint deja un flujo funcional de extremo a extremo: autenticacion, dataset, ventas diarias, deteccion de anomalias y evidencia documentada para la revision.

2. Trabajo del Equipo

  • Juan Pablo Blanco: modelos base de Prisma para Dataset, DailySales y Anomaly, relaciones y middleware requireAdmin.
  • Diego Mercado: CRUD de datasets, proteccion con authenticateToken y control de permisos en borrado.
  • Carlos Moreno: CRUD de ventas diarias, modulo de anomalias, analisis con z-score, email simple, Swagger, README, demo y planeacion de Sprint 4.

3. Arquitectura Mantenida

Se conservo la estructura simple del proyecto para mantener el backend entendible y facil de presentar: 

src/
├── config/
├── controllers/
├── middlewares/
└── routes/
  • Sin capas extra como services, repositories o DTOs.
  • Prisma usado directo en controllers.
  • Rutas protegidas con JWT y permisos basicos de admin en borrado.

7. Flujo de Analisis y Notificacion

  1. El usuario inicia sesion y obtiene su token.
  2. Se crea un dataset para registrar un conjunto de ventas.
  3. Se insertan varias ventas diarias con fecha y monto.
  4. El endpoint de analisis toma las ventas del dataset.
  5. Calcula promedio, desviacion estandar y z-score.
  6. Guarda las ventas atipicas como anomalias.
  7. Si hay configuracion SMTP, envia correo; si no, deja evidencia en consola.
